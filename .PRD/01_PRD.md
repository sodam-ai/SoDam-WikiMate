# Wikimate — PRD (Product Requirements Document)

> 생성일: 2026-06-07 · **버전: v2** (멀티 에이전트 플러그인 + 공식 CLI 반영)
> 생성 도구: Show Me The PRD
> 제품명: **Wikimate (위키메이트)** — "AI 에이전트 자료 정리 키트 (Obsidian × Notion)" · LLM이 읽는 내 지식 위키를 함께 만드는 메이트

---

## 1. 제품 개요

### 한 줄 요약
AI 에이전트(Claude Code·Codex·Gemini)에게 "이거 정리해줘"라고 명령하면, 흩어진 자료(웹·PDF·AI 대화로그·코드)를 읽어 **옵시디언(장기 기억)** 에 노트로 정리하고 **노션(색인·운영판)** 에 표로 색인해주는 **멀티 에이전트 도구**. 핵심은 **이식 가능한 MCP 코어 1개**, Claude Code에선 **마켓플레이스 플러그인**(자동 트리거 skills + 서브에이전트 + 자동 연결)으로 제공.

### 해결하는 문제
- 자료가 브라우저 탭·다운로드 폴더·대화창에 흩어져 **나중에 다시 못 찾는다.**
- 손 정리는 시간이 많이 들어 결국 안 하게 된다.
- **AI가 다시 읽고 활용할 수 있는 구조**가 없다. (프롬프트를 잘 써도 읽을 DB가 없으면 소용없음)
- 도구(Claude/Codex/Gemini)를 갈아탈 때마다 **세팅을 처음부터** 다시 해야 한다.

### 핵심 가치
> "AI를 잘 쓰는 사람은 프롬프트를 저장하는 사람이 아니라, **AI가 다시 읽을 수 있는 DB를 만드는 사람**이다."

- **역할 분담**: 옵시디언=장기 기억 · 노션=운영판/색인 · 에이전트=실행 직원 · MCP=AI용 USB-C 포트.
- **모델 비종속(이번 v2의 핵심)**: 실제 정리 로직을 **MCP 코어**에 한 번 담아 모든 에이전트가 공유. 도구가 바뀌어도 **DB·로직 재사용**. 공통 규칙은 `AGENTS.md`(크로스툴 표준).
- **설치하면 알아서 동작**: 설치 시 MCP 자동 등록 + 설치된 CLI(`obsidian`, `ntn`)/MCP 자동 감지·연결 + skills 자동 트리거 → **일일이 호출 안 해도** 됨.
- **안전이 기본값**: 외부 자료를 강한 권한의 AI가 처리 + 공개 배포까지 → "기능"보다 **안전 게이트**가 척추. (분석→보고→**사람 승인**→실행)

### 제품 형태 — "MCP 코어 1개 + 에이전트별 래퍼"
> 플러그인/마켓플레이스 포맷은 **Claude Code 전용**이고 Codex/Gemini는 메커니즘이 다릅니다. 세 도구의 **유일한 공통분모는 MCP**. 그래서 "플러그인 하나로 전부"가 아니라 아래 구조로 "모든 에이전트 사용 가능"을 정직하게 달성합니다.

| 계층 | 내용 |
|---|---|
| **MCP 코어**(이식 가능) | 정리 로직(수집·분류·요약·연결·노션 색인)을 MCP 도구로 노출 — 모든 에이전트 공통 |
| **Claude Code 플러그인**(플래그십·마켓플레이스) | skills(자동 트리거)+commands+agents(서브에이전트)+hooks(자동 연결)+`.mcp.json` |
| **Codex 어댑터** | `AGENTS.md` + `~/.codex/config.toml`(MCP 등록) + 설치 스크립트 |
| **Gemini 어댑터**(후순위) | `GEMINI.md` + 확장 |
| **공통 규칙** | `AGENTS.md`(크로스툴) |

### 멀티 에이전트 호환 매트릭스 (정직하게)
| 기능 | Claude Code | Codex | Gemini CLI |
|---|---|---|---|
| 설치 형태 | 마켓플레이스 플러그인(풀) | 어댑터(AGENTS.md+config.toml) | 어댑터(후순위) |
| MCP 코어 사용 | ✅ | ✅ | ✅ |
| 자동 트리거 skills | ✅ | ⚠️ AGENTS.md 규칙으로 유사 | ⚠️ |
| 서브에이전트 | ✅ | 제한적 | 제한적 |
| 자동 연결 hook | ✅ | 설치 스크립트 | 설치 스크립트 |

## 2. 사용자

### 주요 사용자
- **누구**: 본인 (비개발자·바이브코더). 옵시디언·노션 사용 중, Claude Code·Codex 일상 사용. (배포하면 같은 워크플로의 다른 바이브코더도 사용자)
- **상황**: 자료는 많이 모으지만 정리가 밀릴 때. 터미널에서 에이전트에 명령 가능.
- **목표**: 흩어진 자료를 "다시 찾고·다시 읽을 수 있는" 개인 지식 DB로. 도구를 갈아타도 그대로.

### 사용자 시나리오
1. Wikimate를 설치한다 (Claude Code: `/plugin install`, Codex: 어댑터 스크립트).
2. 설치 즉시 MCP가 등록되고, hook가 `obsidian`/`ntn` CLI·노션 MCP 설치 여부를 **자동 감지·연결**(없으면 안내).
3. 자료를 옵시디언 `00_Inbox`에 넣고 에이전트에 명령: *"Inbox 정리해줘. 계획 먼저 보고."*
4. (스킬 자동 트리거) 에이전트가 **계획을 보고**하고 멈춘다.
5. 승인하면 → 노트 변환·분류·요약·연결 → 노션 색인(`ntn` 우선) → Run Log 기록.
6. 노션 표 / 옵시디언 그래프뷰로 확인.

| ③ 자동 분류·태그·폴더 | 주제 파악 → 7폴더 + 태그 + frontmatter | P1 (MVP·1b) | 보통 |

## 4. 사용자 흐름

### 핵심 흐름
```
설치 → (자동) MCP 등록 + CLI/MCP 자동 감지·연결
   → 자료를 Inbox에 → 에이전트에 명령(스킬 자동 트리거) → 계획 보고
   → 사람 승인 → 변환·분류·요약·연결 → 노션 색인 + Run Log → 확인
```

### 상세 흐름
1. **설치/연결**: 플러그인 설치 → MCP 자동 등록 → hook가 `obsidian`/`ntn`/노션 MCP 감지·연결(없으면 친절 안내).
2. **투입**: 자료를 `00_Inbox`에 넣거나 링크 전달.
3. **명령**: 자연어 명령 → 관련 스킬 자동 트리거.
4. **계획 보고(게이트)**: "무엇을·어디에·어떻게" 보고하고 멈춤.
5. **승인 → 실행**: 변환→분류→요약→링크·MOC→노션 색인. (대량이면 소스별 서브에이전트 병렬)
6. **기록**: Run Log에 요청·변경 노트·승인 여부.
7. **확인**: 노션 표 / 옵시디언 그래프뷰.

## 5. 성공 기준

- [ ] MCP 코어가 Claude Code·Codex **양쪽에서** 같은 "수집" 도구로 동작한다.
- [ ] 자료 1건(웹 링크)을 명령 한 번으로 옵시디언 노트(.md, frontmatter)로 만든다.
- [ ] 설치 시 `obsidian`/`ntn` CLI·노션 MCP를 **자동 감지**하고, 없으면 깨지지 않고 안내한다.
- [ ] 같은 자료 2번 → **중복 노트 없음**(dedup).
- [ ] 자동 트리거 스킬이라도 **승인 없이는 기존 노트를 안 건드린다.**
- [ ] 노트 1개 ↔ 노션 색인행 1줄이 `notion_id`로 연결돼 점프된다.
- [ ] 외부 자료 속 "이전 지시 무시하고 ~해라" 문장을 **명령으로 실행하지 않는다**(인젝션 방어).
- [ ] (배포 시) 배포물에 **개인 볼트경로·토큰·노트 데이터가 들어 있지 않다.**

## 6. 안 만드는 것 (Out of Scope)

- **콘텐츠 생산(스레드·블로그·강의안 자동 생성)** — "정리"가 아닌 "생산". 목적 이탈.
- **노션 운영판 전체(Task/Content/Prompt DB)** — MVP엔 Research Library+Run Log면 충분. Phase 3 선택.
- **노션→옵시디언 역방향(양방향)** — 충돌·복잡도·OAuth/rate-limit. 단방향 유지.
- **완전 무인 자동화** — `ntn`/노션 MCP OAuth + 안전상 부적합. 자동 트리거는 하되 쓰기는 승인.
- **팀 공유·협업·권한 관리** — 1인 로컬(배포해도 각자 로컬).
- **자체 GUI 앱/플러그인 UI** — 제품 형태는 "에이전트에 명령".
- **미검증 마켓플레이스 등록** — 손 시뮬+보안 검증 전 배포 금지.
